Virtual Threads 才是 Java Web 的正解

坦白說，我一直覺得很多團隊不是系統撐不住，
而是工程複雜度先把人搞垮了。

最經典的例子就是：
👉 把原本運作良好的 Spring MVC，硬生生改成 WebFlux。

理由通常只有一句話：

「Non-blocking，效能比較好。」

但實際上呢？
效能沒有飛起來，Debug 卻直接下地獄。

直到 JDK 21 的 Virtual Threads 出現

Java Web，終於回到一條「人類能走的路」。

Thread 為什麼一直是 Java 的痛？

先講一個最基本、但很多人其實沒真的想清楚的事：

Thread 是「誰在執行你的程式碼」

在傳統的 Java Web 架構裡：

  • 一個 HTTP request
  • 對應一條 Thread
  • Thread 一路跑 Controller → Service → DB

只要遇到 IO（資料庫、外部 API）：

👉 Thread 就會卡住等待

這在過去是個大問題，因為：

  • OS Thread 非常重
  • 能開的數量有限
  • 一卡住就等於浪費資源

於是整個 Java 生態開始「扭曲」，
大家開始不寫人類能看懂的程式碼，只為了躲避 blocking。

二、WebFlux 在解什麼問題？

WebFlux 的出發點其實很單純：

不要讓 Thread 等 IO

它的世界觀是：

  • 少量 Event Loop Thread
  • 所有 IO 都必須 non-blocking
  • 用 Mono / Flux 串起整個流程

理論上看起來很美：

  • Thread 利用率高
  • 高併發沒問題

但現實世界往往是：

  • Stack trace 破碎
  • Debug 極度痛苦
  • 一個 blocking call 就能炸整個系統

大多數業務系統的瓶頸，根本不在 Thread

而是在：

  • DB 本身就慢
  • 外部 API 不穩
  • 商業邏輯複雜

結果常見的結局是：

  1. 系統沒有明顯變快
  2. 工程複雜度卻暴增
  3. 團隊士氣一路下滑

Virtual Threads 改變了什麼？

JDK 21 的 Virtual Threads，只做了一件事，
但這件事非常關鍵：

讓「同步寫法」也能具備「非同步的 scalability」

過去：

  • Blocking = 佔住一條 OS Thread
  • Thread 一多，系統就撐不住

現在：

  • Blocking = JVM 幫你掛 Virtual Thread
  • OS Thread 去跑別的任務

Virtual Thread 的特性很簡單：

  • 非常輕量
  • 由 JVM 管理，不是 OS
  • IO 時自動 park / unpark

你寫的程式碼依然是同步的：

User user = userService.getUser(id);
Order order = orderService.getOrder(user);
return new Result(user, order);
